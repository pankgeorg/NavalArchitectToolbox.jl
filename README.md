From a naval-architecture propeller **section table** (chord, pitch,
thickness, camber, skew, rake vs `r/R`) to **3-D blade geometry** and a
**validated open-water vortex-lattice solver**. NAT reads the table, wraps
each 2-D section onto its cylinder with the correct pitch/skew/rake, and
emits a VLM camber grid, a thick surface mesh, and a signed-distance
function вЂ” and it solves the open-water performance itself.

DTMB 4382 (5-blade, 36В°-skew DTMB modified series) is baked in (`dtmb4382`).

## Open-water VLM вЂ” reproduces DTMB 4382

`openwater_vlm` is a from-scratch key-blade vortex-ring lattice method
(Anevlavi, Zafeiris, Papadakis & Belibassakis, *JMSE* **2023**, 11, 2179).
At the design advance ratio J = 0.889:

| coefficient | NAT VLM | experiment |
|---|---|---|
| KT      | 0.207 | 0.208 |
| 10В·KQ   | 0.444 | 0.445 |
| О·       | 0.660 | 0.661 |

Mesh- and wake-converged. Key details that make or break it:

1. **Helical wake winding *with* the rotation** (the downstream extension of
   the blade surface). A flow-aligned wake at the advance pitch winds the
   opposite way, grazes the control points, and makes the influence matrix
   singular. By default the wake is a **transition wake** (pitch eases from
   the blade's geometric pitch at the TE to mean(geometric, advance)
   downstream вЂ” `wake_pitch=:transition`).
2. **Pressure-normal forces, not KuttaвЂ“Joukowski** вЂ” the ring force is
   О”pВ·AВ·nМ‚ along the *surface normal* (О”c_p = C_LEВ·2В·V_mВ·G/Vв€ћВІ). The
   geometric force direction gives the correct О·; О“Г—V gives О· > 4.
3. **Handedness consistency** вЂ” the relative inflow must run LEв†’TE with the
   camber on the suction side; backwards inflates the AoA ~6Г—.
4. A small **lifting-surface pitch correction** (`pitch_corr=1.9454`, default
   on) subtracts a no-lift-angle term в€ќ (f/c)В·(t/c) from the geometric pitch.

The full reproduction (paper citation, code, figures) lives in
[`papers/anevlavi_belibassakis_2023/`](papers/anevlavi_belibassakis_2023).

## The unified "naval tools" surface

`using NavalArchitectToolbox` now brings up three lifting/circulation

- **Propeller open-water VLM** вЂ” `openwater_vlm`, `dtmb4382` (above).

  VortexLattice.jl, validated against lifting-line theory (slope, parabolic
  induced drag). NAT depends on LiftingSurfaces but stays light otherwise.
- **Flettner-rotor panel method** вЂ” `flettner_panel`, `flettner_analytic`
  (below). Pure Julia (LinearAlgebra + StaticArrays only).

The *viscous* Flettner run (WaterLily) lives in ShipFlow.jl, which has the
WaterLily dependency вЂ” NAT itself never takes a WaterLily dep.

## Why this exists

No existing package turns a 2-D-section propeller table into a 3-D blade:

| Package | Role | Reads this table? |
|---|---|---|
| [VortexLattice.jl](https://github.com/byuflowlab/VortexLattice.jl) | generic VLM solver | no вЂ” wants geometry built |
| [CCBlade.jl](https://github.com/byuflowlab/CCBlade.jl) | BEM (chord/twist + polars) | partially вЂ” no 2-D geometry/skew |
| [ParametricBodies.jl](https://github.com/WaterLily-jl/ParametricBodies.jl) | thin planar membranes | no вЂ” can't do thick twisted blades |

## Findings (ecosystem)

- **Generic VLM (VortexLattice.jl)** is numerically delicate for this
  prop: the multi-blade AIC is singular (near the hub the chord exceeds the
  inter-blade gap, so blades overlap). NAT's own key-blade vortex-ring
  formulation avoids this вЂ” it's why `openwater_vlm` exists.
- **ParametricBodies** `PlanarBody` is a thin planar membrane with fixed
  ~grid-scale thickness (Lauber & Weymouth 2022); it can't represent a
  thick twisted cambered blade вЂ” use `blade_sdf` + WaterLily `AutoBody`.
